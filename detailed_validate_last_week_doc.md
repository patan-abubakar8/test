**Initial Setup & Employee Resolution**
```csharp
public TimesheetTaskPendingHours ValidateLastWeekTimesheetEntry(TimeSheetEntryViewModel model, string UserId)
{
    var retData = new TimesheetTaskPendingHours { IsValidTSEntry = true };
    if (model == null || string.IsNullOrEmpty(UserId)) return retData;

    // Resolve employee id (Internal vs External Customer Employee)
    var empData = unitOfWork.repCompanyBuEmployee.GetAll(c => c.UserId == UserId && !c.IsDeleted).FirstOrDefault();
    int eID;
    if (empData == null)
    {
        var custEmp = unitOfWork.repCustomerBuEmployee.GetAll(c => c.UserId == UserId).FirstOrDefault();
        if (custEmp == null) return retData;
        eID = custEmp.Id;
    }
    else { eID = empData.Id; }
```
The system distinguishes between internal employees (`CompanyBuEmployee`) who work directly for the organization, and external customer employees (`CustomerBuEmployee`). It dynamically fetches the underlying integer ID (`eID`) for Database queries.

**Temporal Bounding & Date Mathematics**
```csharp
    // Determine the baseline date of the current entry being inserted
    DateTime baseStart = (model.StartDatetime != DateTime.MinValue ? model.StartDatetime : model.CapturedDateTime).Date;

    // Find the Monday of the current week (Target entry's week)
    int diff = (7 + (baseStart.DayOfWeek - DayOfWeek.Monday)) % 7;
    DateTime currentStart = baseStart.AddDays(-diff);
    
    // Find the Sunday of the current week
    DateTime currentEnd = currentStart.AddDays(6);
    if (currentEnd < currentStart) { currentEnd = currentStart.AddDays(6); } // Fallback check

    // Calculate the precise bounds of the immediate PREVIOUS week
    DateTime prevWeekStart = currentStart.AddDays(-7).Date;
    DateTime prevWeekEnd = currentEnd.AddDays(-7).Date;

    // Global cut-off period: The validation does NOT enforce anything before November 1st, 2025.
    var cutOffDate = new DateTime(2025, 11, 1);
```
The logic locks down the current target date to its weekly boundaries (Monday to Sunday) using modular arithmetic. From there, it jumps backward exactly 7 days to isolate the "Previous Week".

**Database Fetching & Validation Logic**
Now, the application performs three targeted repository queries against `TimesheetPeriod` and raw `TimesheetEntry`.

```csharp
    // QUERY 1: Check for unapproved entire PERIODS
    // Looks for periods grouped for this employee that ended before the current week started,
    // occurred after the cut-off, and are not in "Approved" state.
    var pendingPeriods = unitOfWork.repTimesheetPeriod.GetAll(p => p.EmployeeId == eID)
        .Where(p => p.EndDate.Date < currentStart &&
                    p.EndDate.Date >= cutOffDate &&
                    p.ApprovalStatus != (int)ApprovalStatusType.Approved)
        .OrderBy(p => p.StartDate)
        .ToList();

    // QUERY 2: Check for unapproved loose ENTRIES in the immediate previous week
    // Covers cases where an entry was created but no period was successfully linked/processed yet
    var prevWeekEntries = unitOfWork.repTimesheetEntry
        .GetAll(e => e.UserId == UserId &&
                     e.CapturedDateTime.Date >= prevWeekStart &&
                     e.CapturedDateTime.Date <= prevWeekEnd &&
                     e.ApprovalStatus != (int)ApprovalStatusType.Approved)
        .ToList();

    // QUERY 3: Check for unapproved loose ENTRIES older than the previous week (but after cut-off)
    var olderEntries = unitOfWork.repTimesheetEntry
        .GetAll(e => e.UserId == UserId &&
                     e.CapturedDateTime.Date < prevWeekStart &&
                     e.CapturedDateTime.Date >= cutOffDate &&
                     e.ApprovalStatus != (int)ApprovalStatusType.Approved)
        .ToList();
```

**Message Construction & Return Result**
If any of these queries return results, the user is violating the policy. The code collects all the violating dates, constructs a descriptive human-readable statement into the `BillType` field, flags `IsValidTSEntry` as `false`, and sums up the total delinquent hours so the UI can contextualize the warning.

```csharp
    if (pendingPeriods.Any() || prevWeekEntries.Any() || olderEntries.Any())
    {
        var pendingWeekStarts = new List<string>();

        // Add start dates for failing periods
        pendingWeekStarts.AddRange(pendingPeriods.Select(p => p.StartDate.ToString("dd/MM/yyyy")));

        // Add start date of previous week if raw entries found but no tracking period exists
        if (prevWeekEntries.Any() && !pendingPeriods.Any(p => p.StartDate.Date == prevWeekStart)) {
            pendingWeekStarts.Add(prevWeekStart.ToString("dd/MM/yyyy"));
        }

        // Add earliest date for older delinquent entries
        if (olderEntries.Any()) {
            var oldest = olderEntries.Min(e => e.CapturedDateTime).Date;
            pendingWeekStarts.Add(oldest.ToString("dd/MM/yyyy"));
        }

        var distinctWeeks = pendingWeekStarts.Distinct().ToList(); 
        
        // Construct detailed warning message for UI
        var message = "Please ensure previous timesheet(s) for the week(s) are submitted and approved before submitting the current period. " +
            "starting: " + string.Join(", ", distinctWeeks);

        // Calculate hours total utilizing LINQ Sum across affected entries
        decimal workedHours = 0;
        if (pendingPeriods.Any()) {
            var periodIds = pendingPeriods.Select(p => p.Id).ToList();
            var entriesForPeriods = unitOfWork.repTimesheetEntry.GetAll(e => periodIds.Contains(e.TimesheetPeriodId)).ToList();
            workedHours += entriesForPeriods.Sum(e => e.Hours);
        }
        if (prevWeekEntries.Any()) { workedHours += prevWeekEntries.Sum(e => e.Hours); }
        if (olderEntries.Any()) { workedHours += olderEntries.Sum(e => e.Hours); }

        retData.IsValidTSEntry = false; // "false" signifies a validation problem exists
        retData.WorkedHours = workedHours;
        retData.TotalHours = 0;
        retData.ValidationType = "PreviousWeekNotSubmittedApproved";
        
        // HACK: Piggybacking the complex error string within the purely informative "BillType" property
        retData.BillType = message; 

        return retData;
    }

    // Success path: return true, system proceeds cleanly
    retData.IsValidTSEntry = true;
    return retData;
}
```

## 6. The Database Layer (Entities & Repositories)

The implementation relies heavily on Entity Framework queries via the Unit Of Work (`unitOfWork`) pattern. The primary tables and specific states queried represent the ultimate ground truth of the validation.

### `TimesheetPeriod`
Groups an entire week's worth of timesheets.
**Schema Reliances**:
- `Id` (Primary Key)
- `EmployeeId` (Foreign Key back to Company/Customer Employee)
- `StartDate` & `EndDate` (Defines the weekly boundary)
- `ApprovalStatus` (Integer enumerator referencing states like `Saved(10)`, `Submitted(11)`, `Approved(12)`, `Rejected(13)`). 
*(The validation explicitly queries for `ApprovalStatus != 12` (Approved) to decide if a past week is delinquent).*

### `TimesheetEntry`
The base recording of time spent on a task. 
**Schema Reliances**:
- `CapturedDateTime` & `StartDatetime` (Saves the exact literal date hours were tracked)
- `UserId` (String Guid mapping to identity)
- `TimesheetPeriodId` (Links the loose entry to its parent week group)
- `Hours` (Decimal tally of work)

The repositories parse these raw DB entries into domain objects, evaluate their statuses via LINQ `.Where()` clauses translated into parameterized SQL, and deliver them to memory for the hour-tally logic.

## 7. Flow Summary Execution

1. **User Action**: The `User` edits their `Task` and `Hours` inputs on the `_TimesheetTableLoad` View and clicks the Save Action icon.
2. **Javascript Compilation**: The `updateTimeEntry()` script validates length/values natively, packages all `<td>` row data via data-attributes into a `JSON` `tsModel`.
3. **AJAX Transport**: The browser fires an asynchronous POST request carrying the `tsModel` to `/Timesheet/TimesheetEntryDetails`.
4. **Controller Ingestion**: Model-binding transforms the `JSON` into a typed `TimeSheetEntryViewModel`. Controller normalizes `DateTime` structures and checks `LockPeriod` permissions.
5. **Service Validation Activation**: The Controller fires `ValidateLastWeekTimesheetEntry()`.
6. **Date Math**: The Service identifies the `Monday` of the current entry, identifies the `Monday/Sunday` of the exact prior week, and establishes the strict `Nov 1, 2025` cutoff.
7. **Database Poll**: Service polls DB to retrieve all `TimesheetPeriods` and standalone `TimesheetEntry` rows predating the current Monday that lack an `Approved` status.
8. **Message Payload**: If violations exist, the Service builds an aggregated string indicating unapproved dates and sets `IsValid=False`. 
9. **Soft Fail Resolution**: The Controller captures the string but decides *not* to halt the database write. The timesheet is inserted (`InsertTimesheetEntry()`).
10. **JSON Return**: The Controller returns success `status: true`, but concatenates `" with warning! || "` + the validation message into the standard response payload.
11. **Toast Notification UI**: The JS `success:` block detects the term `"warning"` in the return message and generates a yellow Toastr alert displaying the dates they still need to close out in previous weeks, despite their current work being saved successfully.
